## 二、项目概述

### 1. 项目简介

名著导读是一个基于 Dify 平台的 AI 智能问答应用，通过大语言模型结合知识库检索，为用户提供经典名著的深度导读服务。用户可通过自然语言提问，获取关于名著的人物分析、情节解读、背景知识等内容。

首发项目为《西游记》，后续计划扩展至《三国演义》《红楼梦》《水浒传》等。

**使用场景**：
- 学生课后辅助阅读经典名著
- 文学爱好者深入学习与研究
- 教育工作者备课参考

### 2. 项目目标

**核心业务目标**：
- 降低名著阅读门槛，让经典文学更易理解
- 提供准确、有据可依的名著知识问答
- 打造可持续扩展的多作品导读平台

**技术目标**：
- 基于 RAG（检索增强生成）架构，确保回答准确可溯源
- 通过条件分支实现不同场景的差异化回复策略
- 支持知识库的持续扩展与更新

### 3. 适用人群

- **最终用户**：学生、文学爱好者、教育工作者
- **开发者**：Dify 工作流维护人员、知识库管理员
- **运维人员**：负责 Dify 平台维护、模型 API Key 管理

### 2. 部署步骤

#### 步骤 1：导入工作流

1. 登录 Dify 平台
2. 进入「工作室」→「导入应用」
3. 选择 `03_工作流/名著导读.yml` 文件
4. 确认导入，应用自动创建

#### 步骤 2：配置模型提供商

1. 在 Dify 设置中进入「模型提供商」
2. 配置 **DeepSeek**：填入 API Key，确认 `deepseek-v4-flash` 模型可用
3. 配置 **SiliconFlow**：填入 API Key，确认以下模型可用：
   - `netease-youdao/bce-embedding-base_v1`（Embedding）
   - `netease-youdao/bce-reranker-base_v1`（Reranker）

#### 步骤 3：创建并配置知识库

1. 在 Dify「知识库」中创建新知识库，命名为「名著导读-西游记」
2. 上传西游记相关文档（原文片段、人物介绍、文学评论等）
3. 选择 Embedding 模型为 `bce-embedding-base_v1`
4. 等待文档向量化处理完成
5. 回到工作流编辑界面，将知识检索节点的数据集 ID 替换为新知识库的 ID

#### 步骤 4：本地环境配置

```bash
# 创建虚拟环境
python -m venv .venv
source .venv/bin/activate  # Linux/Mac
# .venv\Scripts\activate   # Windows

# 安装依赖
pip install -r requirements.txt
```

#### 步骤 5：发布应用

1. 在工作流编辑器中点击「发布」
2. 确认工作流无报错后完成发布
3. 通过 Dify 生成的分享链接或内嵌 Web 页面访问

## 七、接口开发文档

### 统一请求格式

本项目为 Dify 应用，对外接口由 Dify 平台统一提供。以下为 Dify Chat API 标准格式：

**请求方式**：`POST`

**请求头**：
```
Authorization: Bearer {api_key}
Content-Type: application/json
```

**入参格式**：
```json
{
    "inputs": {},
    "query": "孙悟空有哪些经典故事？",
    "response_mode": "streaming",
    "conversation_id": "",
    "user": "user-001"
}
```

### 接口列表

| 接口名称 | 请求地址 | 请求方式 | 说明 |
|----------|----------|----------|------|
| 发送对话消息 | `/v1/chat-messages` | POST | 发送用户问题并获取 AI 回复 |
| 获取会话列表 | `/v1/conversations` | GET | 获取用户的历史会话列表 |
| 获取消息列表 | `/v1/messages` | GET | 获取指定会话的消息记录 |
| 停止响应 | `/v1/chat-messages/{task_id}/stop` | POST | 停止正在生成的 AI 回复 |

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| query | string | 是 | 用户提问内容 |
| inputs | object | 否 | 工作流输入变量（本应用为空） |
| response_mode | string | 是 | `streaming`（流式）或 `blocking`（阻塞） |

## 八、核心业务逻辑

### 1. 整体流程

```
用户输入问题
    │
    ▼
[开始节点] — 接收 sys.query
    │
    ▼
[知识检索] — 在知识库中检索相关文档（Top 4）
│   • BCE Embedding 向量化用户问题
│   • 向量相似度检索
│   • BCE Reranker 重排序
│   • 返回 resultArray
    │
    ▼
[条件分支] — resultArray 是否非空？
    │
    ├─── TRUE（检索命中）──→ [LLM 通用问答]
    │                        │ • 基于上下文严谨回答
    │                        │ • 优先使用原文信息
    │                        │ • 知识库不足时明确说明
    │                        │
    │                        ▼
    │                    [参数提取器] — 提取关键信息
    │                        │
    │                        ▼
    │                    [直接回复] ← 输出参数 a
    │
    └─── FALSE（检索未命中）──→ [LLM 学者风格]
                                 │ • 以四大名著学者身份回答
                                 │ • 半文半白风格
                                 │ • 无法回答时诚实告知
                                 │
                                 ▼
                             [参数提取器] — 提取关键信息
                                 │
                                 ▼
                             [直接回复] ← 输出参数 b
```

### 2. 关键模块执行逻辑

**知识检索模块**：
- 检索模式：混合检索（向量检索权重 1.0 + 关键词检索权重 0）
- Reranking：启用，使用 BCE Reranker 对召回结果重排序
- 返回数量：Top 4 个最相关文档片段
- 分数阈值：未设置（null），全量返回 Top K

**条件分支逻辑**：
- 判断条件：知识检索结果数组（`resultArray`）不为空
- 分支效果：有上下文时走严谨模式，无上下文时走主观解读模式

**双 LLM 策略**：
- 有检索结果 → 知识问答助手角色，严格基于上下文
- 无检索结果 → 四大名著学者角色，允许一定主观性但标注限制

### 3. 数据流转过程

```
用户问题 (string)
  → 向量化 (float[]) 
    → 相似度检索 (document chunks)
      → Reranker 重排序 (ranked chunks)
        → 条件判断 (boolean)
          → Prompt 拼接 (system prompt + context + question)
            → LLM 推理 (text)
              → 参数提取 (string)
                → 回复输出 (string)
```

## 九、功能模块说明

### 模块 1：知识检索问答（主路径）

**功能**：基于知识库检索结果，结合 LLM 生成准确、有据可查的回答。

**使用方法**：用户直接提问，系统自动检索知识库并作答。

**限制条件**：回答质量依赖知识库覆盖度；回答长度受 max_tokens（512）限制。

### 模块 2：学者风格问答（降级路径）

**功能**：当知识库无匹配结果时，以学者身份基于 LLM 自身知识作答。

**使用场景**：用户问题超出知识库范围，但与四大名著相关。

**限制条件**：回答以「此节在演义中并无详细记载」标识知识盲区；不保证 100% 准确性。

### 模块 3：知识库管理

**功能**：管理西游记及后续作品的知识数据，支持文档上传、分段、向量化。

**使用方法**：通过 Dify 知识库界面或 API 上传/更新文档。

**限制条件**：单文件大小受 Dify 平台限制；向量化处理需要时间。

### 模块 4：TTS 原文朗读（V2.2.0 新增）

**功能**：在原著阅读区提供语音朗读功能，用户可听读原文。

**技术方案**：基于浏览器内置 Web Speech API（SpeechSynthesis），无需后端支持。

**交互控制**：
- 播放/暂停/停止：底部控制栏三按钮，状态联动
- 段落高亮：当前朗读段落黄色高亮，视觉聚焦
- 进度条拖动：可拖动进度条跳转至任意段落
- 文本跟随滚动：朗读段落自动滚动至可视区域

**使用方法**：在左栏原著阅读区点击底部控制栏播放按钮开始朗读。

**限制条件**：依赖浏览器 SpeechSynthesis 接口，不同浏览器语音引擎效果有差异；不支持 IE 及部分老旧浏览器。

## 十、常见报错与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 知识检索始终返回空 | 知识库未正确关联或 API Key 失效 | 检查 SiliconFlow API Key；确认数据集 ID 正确；检查知识库是否有已处理的文档 |
| LLM 节点返回超时 | API 限流或网络问题 | 检查 DeepSeek API 余额与限额；考虑增加超时时间 |
| 工作流导入失败 | YML 文件格式不兼容 | 确认 Dify 版本 ≥ 0.5.0；检查 marketplace 插件版本 |
| Embedding 模型不可用 | SiliconFlow 账户未开通该模型 | 登录 SiliconFlow 控制台确认模型权限 |
| 回答质量差、答非所问 | 知识库内容不足或检索参数不当 | 补充知识库文档；调整 Top K 值；优化文档分段策略 |
| 端口冲突 | 本地调试时端口被占用 | 修改 Dify 服务端口配置 |
| API Key 鉴权失败 | Key 已过期或权限不足 | 重新生成 API Key；确认 Key 有对应模型调用权限 |
| 工作流发布后无法访问 | 应用未发布或分享链接未生成 | 在 Dify 中重新发布并生成新的分享链接 |

- `server/middleware/` — JWT 依赖注入、异常处理、日志中间件
- `server/config.py` — 环境变量解析
- 旧版 `serve_frontend.py` 保留作为参考

#### S6-T02 API 限流（后端）

slowapi 实现 6 端点分级限流：
- 登录：5/min + 连续失败锁定
- 注册：10/min
- 对话：20/min
- 搜索：30/min
- 健康检查/章节：60/min

#### S6-T03 安全加固（后端+隐私）

- 安全响应头中间件：X-Frame-Options / X-Content-Type-Options / X-XSS-Protection / Referrer-Policy
- 全局异常捕获：覆盖 404/405/422/500/503
- 隐私数据扫描与脱敏（隐私）

#### S6-T04 日志系统（后端）

双通道日志（`server/logger.py` + `logging.conf`）：
- 控制台实时输出
- 文件滚动：`logs/server_*.log`，按天轮转，保留 30 天
- 请求日志中间件自动记录方法/路径/状态码/耗时
